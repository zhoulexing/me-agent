# Claude Agent SDK 的几个隐藏坑：系统提示词、Skill 召回和上下文压缩

![核心总览图](images/04-summary-2.35.png)

今天是 2026 年 6 月 13 号。

当你用 Claude Code 输入一句你好，然后它回你一句话，就这么一个很短的过程，你知道背后会消耗多少 token 吗？

当你用官方的 Claude Agent SDK，问它你是谁的时候，它有时会告诉你：我是 Claude Agent。

可是你自己的系统提示词里明明没有这句话，甚至你已经强调了角色，它为什么还是会这样回答？

当你在 Claude Agent SDK 里内置了很多 skills，却发现它就是不会召回。

你反复优化 skill 的描述，改得越来越清楚，但效果还是不稳定，这又是为什么？

还有一个更隐蔽的问题：当上下文被压缩之后，你会发现很多 skills 突然找不到了。

明明上一轮还能用，压缩之后就像消失了一样。

今天就聊一下，我们在使用 Claude Agent SDK 和 Claude Code 时，背后很容易踩到的几个坑。

## 第一件事：不要猜模型怎么想，要看它真正看到了什么

要搞清楚这些问题，第一步不是猜模型怎么想，而是要看到它真正调用模型时，请求里到底放了什么。

也就是说，我们要拿到完整的系统提示词、消息列表、工具描述、skill 描述，以及最终返回的响应。

拿到这些内容的方法有很多。

比如你可以自己写一个 AI 网关，把所有模型请求都代理到自己的服务上。这样每一次请求经过网关时，你就能记录完整的 request 和 response。

也可以在本地加一层代理，去监控 Claude Code 或 Claude Agent SDK 最终发出去的 HTTP 请求。

但还有一个更简单的方法：通过配置几个环境变量，让它把每一轮请求体和响应体直接写到本地的 json 文件里。

无论是 Claude Code，还是基于 Claude Agent SDK 写的应用，把这些环境变量加进去，后续每一轮模型调用，它都会把完整的请求体和响应体落到本地文件里。

拿到完整请求之后，很多问题就非常清楚了。

![请求体拆解图](images/01-request-payload.png)

## 第一个坑：你写的系统提示词，不等于模型看到的完整系统提示词

第一个你会发现，系统提示词里本身就内置了一些身份相关的内容。

比如它会告诉模型：你是 Claude Code，或者你是 Claude Agent。

这就解释了为什么很多时候，你明明在 SDK 层面改了角色，甚至在自己的系统提示词里要求它扮演另一个角色，它还是会以 Claude Agent 或 Claude Code 的身份来回答。

因为这些身份信息不是你写进去的，而是框架在更上层提前注入进去的。

更麻烦的是，至少从 SDK 暴露出来的能力看，它没有一个很干净的开关，让你把这部分默认系统提示词直接关掉。

所以如果你真的要彻底控制这部分内容，就只能在 AI 网关层做处理。

比如把最上面的几条系统级消息过滤掉，再转发给模型。

![系统提示词分层图](images/02-system-prompt-layers.png)

## 第二个坑：Skill 召回不准，可能不是描述写得不够好

第二个问题，是 skill 召回为什么会不准。

当你看完整请求时，会发现有些 skill 是有名称和描述的，有些 skill 却只有名称，没有描述。

这就很关键了。

因为大模型判断一个 skill 该不该被调用，很大程度上依赖的就是 name 和 description。

name 只能告诉它这个东西大概叫什么。

description 才能告诉它在什么场景下应该用。

如果 description 根本没有进入上下文，那你把描述写得再好也没用。

模型压根没看到，自然就不可能靠它来做准确召回。

那为什么有些描述会被加载，有些描述不会？

这里背后有一个限制机制。

Claude 对 skill 的加载不是无限制的。它会根据最大上下文窗口按比例计算一个预算，是最大上下文的 1%，再乘以 4。

也就是说，skill 的名称和描述能占用的空间是有上限的。

当你的 skills 特别多时，它不可能把所有 skill 的完整描述都塞进系统提示词里。

于是就会出现一部分 skill 带着 name 和 description 进入上下文，另一部分 skill 只以 name 的形式出现，description 被省略。

更坑的是，这个预算配置可以更改，但是并没有非常明确地暴露在官方文档里。

![Skill 描述预算图](images/03-skill-budget.png)

## 第三个坑：上下文压缩之后，Skill 可能真的会丢

第三个问题，是上下文压缩之后，为什么 skill 会丢。

这件事也要从完整请求里看。

你会发现，整个 skills 的名称和描述，并不是以 system 角色的消息放进去的，而是以 user 角色的消息加进去的。

大模型的消息一般会有 system、user、assistant 这些角色。

system 角色里的内容，通常是更稳定的底层指令。即使上下文压缩，它也会被保留下来，或者至少会被重新注入。

但 user 角色里的内容就不一样了。

它会跟普通对话、工具结果、历史消息一起进入压缩流程。

压缩的目标是保留对当前任务有帮助的信息，而不是保证每一段原始内容都完整保留。

所以当 skill 的名称和描述是以 user 消息形式加入时，一旦上下文压缩，它就有可能被摘要掉、弱化掉，甚至直接丢掉。

这也是为什么很多 Agent 系统在短对话里表现还可以，一进入长任务、长上下文、多轮工具调用，就开始出现能力漂移。

## 真正的结论：Prompt 不是你写了什么，而是最终进了什么上下文

所以这几个问题放在一起，其实都指向同一个核心：

不要只看你写了什么 prompt，要看最终进入模型的上下文到底是什么。

你写的系统提示词，不等于模型看到的完整系统提示词。

你写的 skill 描述，不等于模型每一轮都能看到这些描述。

你以为稳定存在的能力，也可能在上下文压缩之后被挤掉。

如果你正在做 Agent，尤其是基于 Claude Code 或 Claude Agent SDK 做长任务、多工具、多 skill 的系统，这件事就非常关键。

不要只调 prompt。

先把完整请求体抓出来。

很多看起来像模型不听话、skill 不稳定、上下文漂移的问题，最后都会回到同一个地方：

模型到底看到了什么。
