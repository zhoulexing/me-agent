# 基于现有文字稿整理公众号文章

## 适用场景

用户已经提供原始文字稿，并要求整理成公众号文章时使用本流程。

常见输入：

- 口播稿
- 技术认知笔记
- 文章初稿
- 会议/分享整理稿
- 本地 Markdown 文件

目标是保留原文表达，让它更适合公众号阅读和发布，不是重写一篇新文章。

## 核心原则

### 1. 原文保真

默认只做轻量整理：

- 修正错别字、口误和明显病句
- 调整中文、英文、数字之间的空格
- 拆短过长段落
- 增加必要的二级标题和承接句
- 删除明显的视频口播尾巴

不要做这些事：

- 新增原文没有的观点
- 大幅替换原文句子
- 改变原文论证顺序
- 把口语稿改成技术文档
- 为了显得高级而重写表达

如果用户说“改动太多”，回到原文句子，只保留排版、标题、错别字和必要衔接。

### 2. 公众号阅读优先

- 单段要短，每段只讲一个意思
- 只使用 H1 和 H2，默认不要 H3
- 不使用目录、脚注、复杂表格
- 非必要不使用代码块
- 列表不要多层嵌套
- 标题承接阅读节奏，不要过度营销化

### 3. 图片服务内容节点

配图默认用于解释关键节点，不做装饰。

优先位置：

1. 开头：总览图或封面总结图
2. 第一个认知反差之后：解释核心结论
3. 分类/限制段落之后：解释关键框架
4. 流程段落之后：解释方法步骤

图片位置确定后，后续改文尽量不移动。

## 标准流程

### Step 1 读取原文

先完整读取原稿，识别：

- 主题和稿件类型
- 核心问题
- 原文推进顺序
- 适合插图的位置
- 需要保留的日期、模型名、版本号、数值限制

不要没读完就改写。

### Step 2 确定输出目录

在项目内创建独立输出目录。

推荐结构：

```text
outputs/{topic-slug}/
├── wechat-article.md
├── images/
│   ├── 01-xxx.png
│   ├── 02-xxx.png
│   └── 04-summary-2.35.png
└── prompts/
    ├── 01-xxx.md
    ├── 02-xxx.md
    └── 04-summary-2.35.md
```

目录名用英文 kebab-case，建议包含主题和日期。

### Step 3 生成配图提示词

默认使用 `baoyu-slide-deck` 技能的方法生成配图提示词，不需要等用户额外指定。

要求：

- 先按内容节点判断图片数量，默认 4 张以内
- 参考 `baoyu-slide-deck` 的分析、风格选择和生图 prompt 结构
- 只生成用于图片的提示词，不必生成完整 PPT
- 提示词保存到 `prompts/`

常见组合：

- 1 张开头总结横图
- 2 到 3 张正文解释图

提示词必须写清楚：

- 图片比例
- 中文文字
- 核心信息
- 画面布局
- 风格约束
- 不要 logo、页码、页脚
- 信息不要过密

比例建议：

- 正文图：16:9
- 开头总结图：2.35:1

### Step 4 生成图片并落盘

使用可用图片生成能力生成图片。

要求：

- 图片复制到 `images/`
- 原始生成缓存不要删除
- 文件名语义化
- 检查尺寸是否符合要求
- 快速查看图片，确认文字可读、主题准确

关键标题、结论、流程文字错误时，重新生成。

### Step 5 整理公众号结构

只做必要分层，不改变原文主线。

推荐结构：

```markdown
# 文章标题

![开头总结图](images/04-summary-2.35.png)

开头原文或轻微整理后的开头。

## 第一个问题/第一层结论

原文段落。

![正文图一](images/01-xxx.png)

原文段落。

## 第二个问题/第二层结论

原文段落。

![正文图二](images/02-xxx.png)

原文段落。

## 方法/流程/案例

原文段落。

![正文图三](images/03-xxx.png)

原文段落。

## 最后总结

原文总结。
```

标题要求：

- H1 只保留文章标题
- 正文只使用 H2
- H2 要清楚承接原文逻辑
- 不为标题好看改变原文重点

### Step 6 输出公众号适配 Markdown

最终文件默认写入 `wechat-article.md`，内容要能直接进入公众号编辑器继续排版。

Markdown 要求：

- 保留一个 H1
- 正文只使用 H2
- 图片用相对路径
- 图片前后各空一行
- 段落之间空一行
- 单段不要太长
- 不写“以下是公众号文章”
- 不在末尾添加无关说明

排版转换要求：

- 若需要直接复制到公众号，按本 skill 的 `scripts/wechat-richtext-styles.js` 生成公众号富文本/HTML
- 转换后使用内联样式，不依赖外部 CSS、class 或 id
- 基础正文：17px，行高 1.75，颜色 `#333`
- H1：24px，加粗，底部边框，行高 1.5
- H2：20px，加粗，行高 1.5
- 段落：上下 15px 间距
- 列表：保留合理缩进，避免多层嵌套
- 引用：浅灰背景、左侧蓝色边框
- 图片：最大宽度 100%，高度自适应
- 加粗内容统一转换为 `strong` 或 `b`，使用 `font-weight: 700`
- 清理 `class`、`id`、`data-line`、`contenteditable` 等属性

如果用户只要项目落盘，保留干净 Markdown；如果用户要直接粘贴公众号，额外提供转换后的富文本/HTML。

## 交付检查

- 保留了原文核心句子
- 没新增原文没有的观点
- 没改变原文论证顺序
- 没有 H3 及以下标题
- 图片路径能从 Markdown 相对访问
- 总结图在开头
- 正文配图插在对应内容之后
- `prompts/` 中有基于 `baoyu-slide-deck` 方法生成的提示词
- 公众号富文本/HTML 按内联样式转换，适合复制到公众号编辑器

## 用户要求调整

如果用户说“文字内容层面改动太多了”：

1. 回到原文
2. 保留图片位置
3. 只重写 `wechat-article.md`
4. 主要使用原文句子
5. 只保留二级标题、段落拆分、错别字修正和必要空格

如果用户说“更像公众号一点”：

1. 不大幅改观点
2. 优先调整开头、标题和段落节奏
3. 增加少量承接句
4. 原文表达保留 80% 以上

如果用户说“更有传播感”：

1. 可以提炼小标题
2. 可以微调总结金句
3. 仍然不要新增事实或改变技术结论

## 最终回复

完成后简洁说明：

- 文章 Markdown 路径
- 图片目录路径
- 提示词目录路径
- 是否检查过标题层级和图片引用

不要在最终回复里复述整篇文章。
