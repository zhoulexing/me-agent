# 基于现有文字稿整理公众号文章

## 适用场景

当用户已经提供一篇原始文字稿，并要求整理成公众号文章时使用本流程。

典型输入包括：

- 口播稿
- 技术认知笔记
- 已写好的文章初稿
- 会议/分享内容整理稿
- 用户指定的本地 Markdown 文件

这个流程的目标不是重新写一篇文章，而是在尽量保留原文表达的前提下，让内容更适合公众号阅读和发布。

## 核心原则

### 1. 原文保真优先

除非用户明确要求重写，否则不要大幅改动原文观点、论证顺序和句子表达。

允许做的改动：

- 修正明显错别字
- 修正明显口误
- 调整中文和英文、数字之间的空格
- 把过长段落拆短
- 增加二级标题承接内容
- 删除明显的视频口播尾巴，比如“今天的视频就到这里”
- 少量补充承上启下句，但不能改变原意

不应该做的改动：

- 把原文改成全新的表达风格
- 新增原文没有的观点
- 重排到和原文明显不同的逻辑
- 把口语稿改成过度正式的技术文档
- 为了显得更高级而替换大量句子

如果用户反馈“改动太多”，应回到原文句子，只保留排版、标题、错别字和必要衔接。

### 2. 公众号阅读优先

公众号文章不是学术文档，也不是代码文档。

整理时要让读者在手机上能顺着读下去：

- 单段尽量短
- 每段只承载一个意思
- 二级标题要能承接阅读节奏
- 不使用三级及以下标题，除非用户明确要求
- 不使用复杂列表堆叠
- 不使用表格，除非用户明确要求
- 不使用脚注、引用文献式链接和代码块，除非内容本身必须

### 3. 图片服务于内容节点

如果用户要求生成配图，配图不是装饰，而是帮助读者理解关键节点。

优先选择这些位置：

1. 开头：总览图或封面总结图
2. 第一个认知反差之后：解释核心结论
3. 限制/分类段落之后：解释两类限制或关键框架
4. 流程段落之后：解释处理链路或方法步骤

图片位置一旦确定，后续修改文字时应保持不变，除非用户明确要求移动。

## 标准流程

### Step 1 读取原文

先读取用户提供的原始稿。

需要识别：

- 主题是什么
- 原文属于口播稿、文章稿还是笔记
- 核心问题有几个
- 原文的自然段落和推进顺序
- 哪些地方适合插图
- 是否有日期、模型名、版本号、数值限制等可能需要保留的信息

不要在没有读完原文的情况下直接重写。

### Step 2 确定输出目录

在项目内创建独立输出目录。

推荐结构：

```text
outputs/{topic-slug}/
├── wechat-article.md
├── images/
│   ├── 01-xxx.png
│   ├── 02-xxx.png
│   └── 04-summary-2.35.png
└── prompts/
    ├── 01-xxx.md
    ├── 02-xxx.md
    └── 04-summary-2.35.md
```

目录名用英文 kebab-case，包含主题和日期更好。

### Step 3 生成配图提示词

如果用户要求基于内容生图，先生成提示词，再生成图片。

配图数量默认控制在 4 张以内。

常见组合：

- 1 张开头总结横图
- 2 到 3 张正文解释图

提示词要写清楚：

- 图片比例
- 中文文字
- 核心信息
- 画面布局
- 风格约束
- 不要 logo、页码、页脚
- 不要过多信息

正文图常用比例：

- 16:9，适合正文信息图

总结图常用比例：

- 2.35:1，适合公众号开头横图

如果用户指定调用 `baoyu-slide-deck`，应参考它的风格和 prompt 规范，但不必真的生成完整 PPT。可以只借用它的分析、风格选择和生图提示词结构。

技术解释类文章默认优先考虑：

- editorial-infographic
- blueprint
- minimal

### Step 4 生成图片并落盘

使用可用的图片生成能力生成图片。

要求：

- 图片生成后复制到项目输出目录的 `images/`
- 原始生成缓存不要删除
- 文件名必须语义化
- 检查图片尺寸是否符合要求
- 至少快速查看图片，确认没有明显不可读文字或严重跑题

如果图片里的文字有小瑕疵，但不影响理解，可以继续。若关键标题、核心结论、流程文字错误，应重新生成。

### Step 5 整理公众号结构

结构调整只做必要分层，不改变原文主线。

推荐结构：

```markdown
# 文章标题

![开头总结图](images/04-summary-2.35.png)

开头原文或轻微整理后的开头。

## 第一个问题/第一层结论

原文段落。

![正文图一](images/01-xxx.png)

原文段落。

## 第二个问题/第二层结论

原文段落。

![正文图二](images/02-xxx.png)

原文段落。

## 方法/流程/案例

原文段落。

![正文图三](images/03-xxx.png)

原文段落。

## 最后总结

原文总结。
```

标题命名原则：

- 用二级标题承接原文逻辑
- 标题要清楚，不要过度营销化
- 不要为了标题好看而改变原文重点
- 不要使用三级标题

### Step 6 输出公众号适配 Markdown

最终 Markdown 要适配公众号编辑器。

格式要求：

- 文件名默认用 `wechat-article.md`
- 保留一个 H1 作为文章标题
- 正文只使用 H2
- 图片用相对路径
- 图片前后各空一行
- 段落之间空一行
- 单段不宜太长
- 不使用目录
- 不使用脚注
- 不使用复杂表格
- 不使用代码块包裹全文
- 不在正文里写“以下是公众号文章”
- 不在文章末尾添加无关说明

如果用户要直接复制到公众号，优先输出干净正文，不要带产物说明。

如果是在项目中落盘，可以在最终回复里说明文件路径，但不要把路径写进文章正文。

## 保真改写检查清单

交付前检查：

- 是否保留了原文核心句子
- 是否没有新增原文没有的观点
- 是否没有改变原文论证顺序
- 是否只做了必要标题和段落整理
- 是否没有三级及以下标题
- 图片位置是否保持用户要求的位置
- 图片路径是否能从 Markdown 文件相对访问
- 总结图是否在开头
- 正文配图是否插在对应内容之后
- Markdown 是否能直接进入公众号编辑器继续排版

## 用户要求调整时的处理

如果用户说“文字内容层面改动太多了”：

1. 立即回到原文
2. 保留图片位置不变
3. 只重写 `wechat-article.md`
4. 使用原文句子作为主要内容
5. 只保留二级标题、段落拆分、错别字修正和必要空格

如果用户说“更像公众号一点”：

1. 不要大幅改观点
2. 优先调整开头、标题和段落节奏
3. 增加少量承接句
4. 控制在原文表达的 80% 以上

如果用户说“更有传播感”：

1. 可以加强标题和开头钩子
2. 可以提炼小标题
3. 可以微调总结金句
4. 仍然不要新增事实或改变技术结论

## 最终回复

完成后只需要简洁说明：

- 文章 Markdown 路径
- 图片目录路径
- 提示词目录路径
- 是否检查过标题层级和图片引用

不要在最终回复里复述整篇文章。
