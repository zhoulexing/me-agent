# 选题规划

## 目标

这个 reference 用于后续持续找选题，不是一次性头脑风暴。

核心任务有两个：

1. 通过联网搜索，判断最近什么话题值得讲
2. 把这些信号组织进稳定的三类内容体系里

补充要求：

选题不是浅层搜一下就结束，必须做深度研究。深度研究的意思是，不只看什么火，还要看为什么火，谁在讨论，讨论卡在哪，哪些内容已经被讲烂了，哪些角度还有空间。

所有选题默认只围绕三类展开：

1. 分享型内容
2. 解决问题型内容
3. 系统解释型内容

默认占比：

- 分享型：40%
- 解决问题型：20%
- 系统解释型：40%

第三类效果最好，是长期主打。

---

## 三类内容的判断标准

### 1. 分享型内容

适合讲：

- 原则
- 清单
- 方法框架
- 启发总结

特点：

- 好传播
- 好收藏
- 好转发
- 标题要有数字感和结论感

常见形式：

- 3 个
- 5 条
- 7 个摩擦点
- 8 大原则

### 2. 解决问题型内容

适合讲：

- 真实难题
- 设计取舍
- 性能与成本问题
- 上下文、自动化、数据量、稳定性问题

特点：

- 必须具体
- 必须真实
- 必须能复盘

### 3. 系统解释型内容

适合讲：

- 一个系统是怎么运作的
- 一个概念为什么会出现
- 一条完整链路是怎么串起来的

特点：

- 最容易形成代表作
- 最能建立专业标签
- 适合做长线系列

补充判断：

- 最好能解释一个新范式为什么成立
- 最好能解释一个旧范式为什么开始失效
- 最好同时包含机制、边界和代价
- 最好不是只讲结论，而是能讲清背后的结构变化

---

## 研究型好选题标准

遇到论文、技术报告、研究博客时，不要只判断新不新，要判断它配不配进入后续选题池。

优先保留这类研究：

1. 不是单点技巧，而是在重写一个常见问题的理解方式
2. 不只是提出方法，还明确给出适用边界、失败条件或成本拐点
3. 能从研究问题自然过渡到产品、工作流或行业判断
4. 能和真实创作场景连接，能回答为什么现在值得讲
5. 有机会沉淀成系列，而不是只能做一次热点解读

反过来，下面这些默认不算优先级高的好题：

- 只是又一个 benchmark 提升
- 只是堆模型、堆数据、堆工程 tricks
- 只对论文作者自己的设定成立，迁移价值弱
- 只能讲结论，讲不出结构和方法论

可以直接用下面 6 个问题判断一篇研究值不值得进入题库：

1. 它到底改写了什么旧共识
2. 它提出的是能力增量，还是认知框架增量
3. 它有没有把成功条件和失败条件一起讲清楚
4. 它能不能被翻译成创作者和从业者关心的现实问题
5. 它有没有明显的第二层话题，而不是只能停在论文摘要
6. 它适不适合扩成系列内容

默认优先级：

- 能改写认知框架的研究，优先级最高
- 能解释系统边界和失败机制的研究，优先级很高
- 单纯效果更强但解释价值弱的研究，优先级靠后

## 联网搜索要求

只要用户是在找选题、问最近该讲什么、要做内容规划，就必须联网搜索。

不能只靠已有记忆，也不能只看单一平台。

同时必须做深度研究，不能停留在表层关键词和热点标题。

默认搜索平台：

1. 小红书
2. 抖音
3. 公众号
4. X
5. arXiv

如果用户主题明显偏技术，可额外增加官方文档、论文、项目主页；但上面 5 类平台仍然优先。

---

## 每个平台主要看什么

### 小红书

重点看：

- 最近高频出现的表达方式
- 标题包装方式
- 哪类内容更容易被收藏
- 用户在评论区反复问什么

适合提取：

- 分享型选题
- 面向普通用户的系统解释型切口

### 抖音

重点看：

- 高传播切口
- 冲突感和信息差
- 哪类问题更容易被短视频讲清

适合提取：

- 分享型选题
- 解决问题型选题的开场角度

### 公众号

重点看：

- 更成体系的论证
- 哪些主题已经开始被深度讨论
- 适合做长文和视频展开的框架

适合提取：

- 系统解释型选题
- 分享型里的框架类选题

### X

重点看：

- 海外在讨论什么新概念
- 哪些工具、机制、工作流正在形成共识
- 哪些说法已经开始流行

适合提取：

- 系统解释型选题
- 分享型里的新概念总结

### arXiv

重点看：

- 最近冒出来的新方向
- 有没有值得讲清楚的底层机制
- 哪些技术方向已经从研究走向产品化
- 有没有论文在回答“什么时候有效，什么时候失效”
- 有没有论文在重写行业里默认成立的做法
- 有没有研究能直接升级成系统解释型代表作

适合提取：

- 系统解释型选题
- 少量解决问题型选题的技术依据
- 少量高质量分享型内容里的方法框架与判断标准

注意：

- arXiv 不是直接拿来追热点的
- 它更适合提供“底层趋势”和“概念支撑”
- 对 arXiv 论文的筛选，优先看框架价值，不优先看一点点指标优势

## 研究型案例标准

后续遇到类似论文，可以直接拿下面这个案例标准去比。

### 案例：When Single-Agent with Skills Replace Multi-Agent Systems and When They Fail

为什么这是一个好案例：

1. 它不是在讲一个局部技巧，而是在改写一个高频话题：
   单智能体加技能，什么时候可以替代多智能体
2. 它不是只讲能做什么，也讲什么时候会失效：
   技能库变大后会出现选择退化，甚至出现类似相变的阈值
3. 它给的不是纯工程经验，而是带认知科学框架的解释：
   把技能选择问题和容量限制、语义混淆、层级路由联系起来
4. 它天然适合做系统解释型内容：
   可以讲范式切换、讲成本结构、讲能力边界、讲工程含义
5. 它也能外溢出分享型内容：
   比如多智能体和技能系统的 3 个关键差别，技能库设计的 5 条原则

这类论文为什么值得优先跟：

- 它能连接研究、产品和内容表达三个层面
- 它自带冲突：
  不是简单说多智能体好或不好，而是说替代发生在什么条件下
- 它自带边界：
  不只给乐观叙事，也给失败条件
- 它适合做系列：
  可以拆成范式、机制、容量阈值、分层路由、产品设计五个方向

以后把类似内容纳入优先题库时，默认满足下面至少两个条件：

1. 讨论的是范式替代，不只是方法优化
2. 讨论的是失效边界，不只是成功案例
3. 能从研究推到实际工作流或产品设计
4. 能形成系统解释型长内容，而不是只能做快讯

## 选题工作流

### 第一步：先定范围

先判断用户要的是哪一种：

- 最近值得讲的题
- 某个主题下的题
- 一个月或一季度的规划
- 某个平台适合发的题

### 第二步：联网搜索

至少覆盖多个平台，不要只看一个来源。

搜索时重点记录：

- 最近一周到一个月反复出现的话题
- 标题中反复出现的关键词
- 评论区反复出现的追问
- 文章或帖子里的核心矛盾
- 能否回收到三类框架里

### 第二步之后：做深度研究

在完成基础搜索后，继续往下判断：

1. 这个话题为什么在最近被反复讨论
2. 不同平台讨论的角度有没有差异
3. 用户真正关心的是概念本身，还是背后的实际问题
4. 现在已有内容大多停留在哪一层
5. 还有哪些角度没有被讲透
6. 这个题能不能从热点话题上升成长期题

如果这几步没有做，默认不算完成选题研究。

### 第三步：做归类

把搜索结果整理进三类：

- 分享型
- 解决问题型
- 系统解释型

如果某个题既能做分享，也能做系统解释，优先归到系统解释型。

### 第四步：筛题

每个题都问四个问题：

1. 这个题有没有真实需求
2. 这个题你有没有足够判断
3. 这个题能不能讲出差异
4. 这个题适不适合沉淀成系列

再额外补两个问题：

5. 这个题有没有经过深度研究，而不是只来自表层热度
6. 这个题的最佳切口，是热点切口、问题切口，还是系统解释切口

如果题目来自论文或研究博客，再额外问四个问题：

7. 这项研究是在解释新范式，还是只是在优化旧流程
8. 它有没有明确的失效条件、容量边界或使用代价
9. 它能不能被翻译成非论文读者也会关心的问题
10. 它值不值得作为后续判断类似研究的参考样板

### 第五步：输出

输出时优先给用户：

- 推荐题单
- 每题属于哪一类
- 为什么值得讲
- 建议标题方向
- 相关平台信号来自哪里
- 如果是研究型题目，再补一句：
  它属于“新范式 / 边界解释 / 工作流重写 / 失败机制”中的哪一种

---

## 输出格式

默认按下面格式输出：

### 推荐题单

1. 题目
   类别：分享型 / 解决问题型 / 系统解释型
   为什么值得讲：一句话
   信号来源：平台名 + 简短说明
   建议切口：一句话

如果用户要长期规划，再补：

- 10 题循环建议：4 条分享型、2 条解决问题型、4 条系统解释型

如果用户要更快开写，再补：

- 每个题给 3 个标题候选

## 默认种子题库

当联网搜索后的信号和你的长期方向一致时，可以优先从下面这些题里挑。

### 分享型种子题

1. 分享三个给文章配图的 SKILL
2. CLI 设计的 8 大原则
3. 企业落地 AI 的 7 个摩擦点，以及对个人的 3 点启发
4. 我判断一个 AI 工具值不值得长期用的 5 个标准
5. 做 Agent 时最容易犯的 6 个设计错误
6. 一个好 SKILL 应该满足的 5 个标准
7. 我现在最常用的 10 个 AI 工具与明确分工
8. AI 工作流里最值得固化的 7 个环节
9. 为什么很多人用了 AI，效率反而没起来：5 个根因
10. 从 Prompt 到 Context，再到 Harness：3 个阶段的变化
11. 值得长期投入的 5 类 AI 能力，而不是 50 个零散技巧
12. 我做内容和做工程时，最常复用的 8 个 AI 协作模式

### 解决问题型种子题

1. CLI 和 SKILL 在设计时，如何处理超大数据量问题
2. Claude Code 的 bash 工具，为什么能处理超大内容
3. 一个 Agent 为什么会越跑越偏：上下文污染是怎么发生的
4. 浏览器自动化为什么经常不稳定：到底卡在哪几层
5. AI 工作流里，什么时候该缓存，什么时候该实时拉取
6. 工具很多但效果一般，本质上是信息流没设计好

### 系统解释型种子题

1. 一文彻底了解浏览器自动化
2. Harness Engineering 到底是什么
3. 一个视频彻底搞懂 AI 是如何生图的
4. 从用户发出请求开始，一个 Agent 到底是怎么工作的
5. SKILL、Tool、Read、Bash 之间到底是什么关系
6. 为什么 Context Engineering 不是简单的“塞更多上下文”
7. MCP 到底解决了什么问题，它和传统接口调用有什么区别
8. 一个 AI 编码助手，为什么需要渐进式加载能力
9. 多工具协同的本质：不是工具强，而是接力顺不顺
10. 为什么 Prompt Engineering 的时代正在被 Context Engineering 替代
11. 浏览器、CLI、文档、搜索，这些能力是怎么被 AI 串起来的
12. 大模型为什么既像推理引擎，也像调度器

---

## 约束

- 不要为了凑热点牺牲长期主线
- 不要把选题做成泛泛的 AI 新闻摘要
- 不要只给题目，不解释为什么值得讲
- 不要脱离三类框架无限发散
- 不要把深度研究偷换成多搜几个链接
- 涉及“最近在讨论什么”，必须说明是从哪些平台观察到的
