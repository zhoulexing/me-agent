# Harness Engineering

## 这个选题为什么值得长期做

这个题真正值得讲的，不是一个新名词，而是一个更深的问题：

为什么大模型本身并不可靠，但一套外部结构却能把它约束成一个相对可控的系统。

所以这个题不能只讲：

- 什么是 Harness Engineering
- Harness Engineering 和 Context Engineering 的区别

还必须继续往下讲：

- 为什么 Harness Engineering 能让大模型被控制在一个范围内
- 它到底是通过哪些部件来控制的
- 为什么这些部件有效
- 它的边界又在哪里

这才是长期研究型内容的价值所在。

## 这个题的核心判断

这一条内容最核心的判断是：

Harness Engineering 的本质，不是增强模型，而是限制模型的自由度，让它在一个被设计过的工作环境里行动。

如果再压缩成一句更适合长期使用的话：

Prompt 在影响模型怎么想，Context 在影响模型看见什么，而 Harness 在决定模型整个系统怎么跑。

## 这条内容真正要回答的四个问题

### 1. 为什么大模型需要被控制在一个范围内

因为大模型进入真实环境后，默认问题往往不是不会做，而是：

- 自由度太高
- 容易跑偏
- 容易被噪音污染
- 容易把局部合理扩散成全局错误
- 会把能做误当成该做
- 会把语言上的完成误当成系统层面的完成

所以大模型真正需要的，不只是更强推理，而是一套能限制它乱跑、乱做、乱扩散的外部结构。

### 2. Harness Engineering 到底在控制什么

Harness Engineering 不是在控制模型脑子里具体想什么，而是在控制这些更关键的东西：

- 它能看到什么
- 它能做什么
- 它什么时候做
- 它做完之后怎么被检查
- 它犯错之后怎么被拉回
- 它能不能把局部错误扩散成系统性错误

### 3. Harness Engineering 靠哪些部件来控制

可以先收敛成六个部件。

#### 1. 上下文管理

控制它看到什么，不让信息无限涌入。

#### 2. 工具边界

控制它能做什么，不给它无限动作空间。

#### 3. 权限与规则

控制它哪些事不能碰，哪些事必须满足前置条件。

#### 4. 反馈回路

控制它如何知道自己错了，而不是一直在错误路径上继续跑。

#### 5. 验证机制

控制输出是否真的可交付，而不是只是说自己做完了。

#### 6. 任务分解与隔离

控制复杂任务不要在一个脏上下文里持续堆积，避免污染整个系统。

### 4. 为什么这些部件有效

这部分是这条内容最值钱的地方，因为重点不是列部件，而是解释为什么这些部件真的能约束模型。

#### 上下文管理为什么有效

因为模型不是信息越多越强，而是越容易被当前窗口支配。

它对最近进入窗口的信息、当前可见的信息、被高频重复的信息都非常敏感。一旦上下文被污染，后续判断就会一起变形。

所以上下文管理本质上是在控制模型的注意力环境。

#### 工具边界为什么有效

因为很多错误并不是推理错误，而是动作空间过大导致的误操作。

模型一旦拥有太多工具、太宽的操作范围、太模糊的动作描述，就更容易在错误的地方调用错误的能力。

工具边界本质上是在压缩它的行动空间。

#### 权限与规则为什么有效

因为模型没有稳定的内生风险偏好。

它不会像成熟工程师那样天然知道哪些地方最好别碰，哪些动作虽然能做但风险过高。

所以外部权限和规则，通常比指望模型自觉更可靠。

#### 反馈回路为什么有效

因为模型不会天然从现实结果中自动校准自己。

如果系统没有把失败、冲突、异常、测试错误这些信号清晰地喂回来，它就可能一直停留在自我感觉良好的状态里继续乱跑。

反馈回路本质上是在把现实约束重新接回推理过程。

#### 验证机制为什么有效

因为语言上的完成，不等于系统层面的完成。

模型很容易生成一个看起来像完成的答案，但真实世界要的是能运行、能通过检查、能符合约束、能真正交付。

验证机制本质上是在把真假完成区分开。

#### 任务分解与隔离为什么有效

因为上下文一旦被污染，模型后面的判断通常不是局部变差，而是整体一起变差。

把复杂任务拆开，并把子任务隔离在更小、更干净的上下文里，本质上是在阻止错误传播。

任务分解与隔离不是为了形式上的分工，而是为了控制污染范围。

## 这条内容最终要讲成什么

这条内容真正该讲成：

为什么 Harness Engineering 能让一个本来开放、发散、容易跑偏的大模型，变成一个相对可控、可验证、可交付的工作系统。

所以这条内容不要停留在概念介绍，而是要上升到系统解释。

## 标题方向

不要只写：

- 什么是 Harness Engineering

更适合的标题方向：

- Harness Engineering 真正控制的不是模型能力，而是模型的自由度
- 为什么 Harness Engineering 能把大模型约束在一个可控范围里
- 大模型为什么一旦离开 Harness 就容易失控
- Harness Engineering 真正有效的，不是技巧，而是这 6 个控制部件

## 适合这条内容的结构

### 第一部分：先讲问题

为什么模型一旦进入真实环境，就不能只靠 prompt。

### 第二部分：再讲核心判断

Harness 的本质是限制自由度，不是增强智力。

### 第三部分：再拆六个控制部件

每个部件都讲两件事：

- 它控制什么
- 它为什么有效

### 第四部分：最后讲边界

Harness 不是让模型变正确，而是让错误更早暴露、更难扩散、更容易纠偏。

## 这条内容的边界和结尾判断

Harness Engineering 的价值，不是把模型变成绝对正确的系统，而是让一个本来很开放、很不稳定的模型系统，逐步变成：

- 更可控
- 更可验证
- 更可交付
- 更容易发现错误
- 更不容易把小错误扩散成大事故

最后可以落到这个结论：

Harness Engineering 真正控制的，不是模型会不会想，而是模型能不能在一个被设计好的环境里稳定做事。

## 按天拆分的研究计划

下面这部分不是写作计划，而是研究计划。

目标是把这个题拆成每天都能推进的小块，而不是一次性啃完。

建议先按 12 天跑一轮。

### 第 1 天：定义问题

研究目标：

- 先把选题真正研究的问题写清楚
- 不急着定义 Harness，先定义为什么需要 Harness

当天重点：

- 大模型在真实环境里最常见的失控方式有哪些
- 这些失控到底是模型能力问题，还是系统环境问题
- 为什么只靠 prompt 不够

当天输出：

- 一页问题定义
- 3 到 5 个典型失控场景

### 第 2 天：梳理概念边界

研究目标：

- 讲清 Prompt Engineering、Context Engineering、Harness Engineering 三者边界

当天重点：

- Prompt 主要影响什么
- Context 主要影响什么
- Harness 主要影响什么
- 三者关系是什么，不是什么

当天输出：

- 一版三者差异表
- 一句最适合反复使用的核心定义

### 第 3 天：研究上下文管理

研究目标：

- 单独研究上下文管理这个部件

当天重点：

- 为什么上下文污染会导致模型跑偏
- 为什么不是信息越多越好
- 动态加载、裁剪、压缩、隔离分别在解决什么问题

当天输出：

- 上下文管理控制什么
- 上下文管理为什么有效
- 对应案例 2 到 3 个

### 第 4 天：研究工具边界

研究目标：

- 单独研究工具系统为什么会决定模型稳定性

当天重点：

- 工具太多、太宽、太模糊，会导致什么问题
- 什么叫好的工具边界
- 工具描述、输入输出约束、可见性控制分别在起什么作用

当天输出：

- 工具边界控制什么
- 工具边界为什么有效
- 正反案例各 1 到 2 个

### 第 5 天：研究权限与规则

研究目标：

- 单独研究权限系统、约束规则、执行边界

当天重点：

- 为什么模型不能靠自觉
- 哪些规则应该提前写死
- 哪些动作必须审批，哪些动作可以自动做
- 权限为什么是 Harness 的一部分，而不是附属品

当天输出：

- 权限与规则控制什么
- 权限与规则为什么有效
- 高风险动作清单

### 第 6 天：研究反馈回路

研究目标：

- 单独研究错误信号如何回流给模型

当天重点：

- 为什么成功应该尽量沉默，失败应该清晰发声
- 错误信息怎么给，模型最容易修
- 哪些反馈是有效反馈，哪些反馈只是噪音

当天输出：

- 反馈回路控制什么
- 反馈回路为什么有效
- 有效反馈与无效反馈对照表

### 第 7 天：研究验证机制

研究目标：

- 单独研究什么叫真的完成

当天重点：

- 语言完成和系统完成的差别
- 测试、lint、类型检查、人工确认分别在验证什么
- 为什么很多 Agent 问题不是不会做，而是误判自己已经完成

当天输出：

- 验证机制控制什么
- 验证机制为什么有效
- 一版完成定义清单

### 第 8 天：研究任务分解与隔离

研究目标：

- 单独研究为什么复杂任务必须拆分和隔离

当天重点：

- 子任务为什么不只是为了并行
- 为什么隔离上下文可以防止污染
- 单智能体、子智能体、技能系统分别适合什么场景

当天输出：

- 任务分解与隔离控制什么
- 任务分解与隔离为什么有效
- 适用边界与失败边界

### 第 9 天：找真实案例

研究目标：

- 把前面六个部件放回真实系统里看

当天重点：

- Claude Code
- Codex / CLI agent
- browser automation agent
- 你自己仓库里的 SKILL、bash、read、规则链路

当天输出：

- 至少 3 个真实案例
- 每个案例里哪些部件在起作用

### 第 10 天：研究失败案例

研究目标：

- 不只看成功系统，还要研究失败系统

当天重点：

- 哪些 Agent 看起来很强，但真实环境容易翻车
- 它们是在哪个 Harness 部件上失效了
- 失败到底是模型问题，还是环境设计问题

当天输出：

- 3 个失败案例
- 每个案例对应的失效部件

### 第 11 天：收束成完整论证

研究目标：

- 把前面分散的研究拼成一条完整论证链

当天重点：

- 从问题出发，到核心判断，再到六个部件，再到边界
- 检查有没有只是列点，没有真正论证
- 检查每个部件是不是都回答了控制什么、为什么有效

当天输出：

- 一版完整大纲
- 一版核心论证链

### 第 12 天：生成内容版本

研究目标：

- 把研究稿转成可发布内容

当天重点：

- 选择表达形式：长文、口播、图文、系列
- 控制篇幅，不要一次讲太满
- 先做一个主版本，再拆系列版本

当天输出：

- 主稿标题
- 结构提纲
- 系列拆分方案

## 每天研究时都要反复问自己的问题

为了避免研究越做越散，每天都统一过这几个问题：

1. 我今天研究的这个部件，到底控制的是什么
2. 它为什么真的有效，而不是看起来合理
3. 它解决的是哪一类失控问题
4. 它的边界在哪里，什么时候会失效
5. 它和其他部件是什么关系
6. 我有没有找到真实案例支撑，而不是只靠抽象定义

## 这条研究线最终可以延展的系列题

这个题后面还可以自然拆成一组系列：

- 为什么 Prompt 已经不够了
- 为什么 Context Engineering 不是简单塞更多信息
- Agent 真正需要的不是更多工具，而是更好的工具边界
- 为什么权限系统是 Agent 的核心部件
- 为什么很多 Agent 失败，不是模型差，而是反馈回路太差
- 为什么验证机制决定一个智能体能不能交付
- 子智能体真正重要的不是分工，而是隔离污染

## 最终提醒

做这个题时，最容易犯的错误不是讲错，而是讲浅。

如果最后只是停在：

- Harness 是什么
- Harness 和 Context 有什么区别

那这条内容就还是停留在概念介绍。

真正有价值的部分，是继续追问：

- 为什么它能控制模型
- 它到底在控制什么
- 这些控制为什么有效
- 这些控制的代价和边界又是什么

只有讲到这里，这个题才会变成真正长期的、研究型的、有价值的内容。
