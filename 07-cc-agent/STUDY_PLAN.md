# zlx-cc 学习计划

## 原则

- 保持参考工程的外层职责划分不变，先搭骨架，再补能力。
- 每一步都要求可运行、可验证、可解释。
- 优先建立稳定接口，再向里面填复杂实现。
- 设计上优先考虑：感知环境、干跑优先、退出码控制、防住幻觉、幂等。

## 阶段拆解

### 0. 骨架初始化

目标：

- 建立独立子工程
- 固定目录边界
- 定义入口和命令注册表

完成标准：

- `src/main.js` 能接收命令行参数
- `src/commands.js` 能统一注册命令

### 1. 最小 CLI

目标：

- 支持 `--help`、`-h`
- 支持 `help` 子命令
- 支持未知命令错误输出和退出码

完成标准：

- `node ./bin/zlx-cc.js --help` 成功
- `node ./bin/zlx-cc.js not-found` 返回非 0

### 2. 启动与上下文层

对齐参考：

- `main` 负责入口编排
- `setup` 或 `entrypoints` 负责初始化
- `context` 负责收集运行上下文

目标：

- 加入启动流程
- 收集 cwd、平台、Node 版本、是否 git 仓库等上下文

### 3. 命令系统

对齐参考：

- 命令注册表集中管理
- 命令对象统一定义名称、描述、执行器

目标：

- 增加 `version`、`doctor`、`context` 等命令
- 让帮助页自动从注册表生成

### 4. 工具系统

对齐参考：

- 单独的 `tools/` 目录
- 工具有统一 schema、权限和执行接口

目标：

- 先做最小 `bash`、`read-file`、`write-file` 工具
- 执行前支持 dry-run

### 5. 状态与会话

目标：

- 建最小状态容器
- 区分一次性命令执行和后续可扩展的 REPL/会话

### 6. 权限与安全控制

目标：

- 工具调用前做权限检查
- 支持默认、只读、干跑等模式
- 明确退出码语义

### 7. Query Engine 外壳

目标：

- 先只做引擎接口和循环骨架
- 不急着接复杂模型能力

## 当前范围

本轮只完成阶段 0 和阶段 1。
