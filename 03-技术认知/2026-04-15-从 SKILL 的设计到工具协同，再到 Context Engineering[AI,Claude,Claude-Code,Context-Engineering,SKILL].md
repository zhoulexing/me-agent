今天是2026年的4月15号。

大模型和外界交互，其实只有一种方式，就是基于 tool。

那我们平常写的 skill 到底是怎么被调用的？又是怎么进行渐进式加载的呢？

去年下半年的时候，因为一个项目需要基于 opencode 做一些魔改，所以我专门看过它的设计。前段时间 Claude Code 的源码也泄漏了，今天又稍微看了下，实现方式不一样，但思路都差不多。

核心是有一个专门的 skill 工具，它会先找到用户级和项目级下的所有 skills，然后读取每个 SKILL.md 里的 name 和 description。这样一来，大模型一开始就知道现在有哪些能力可用，但还不会一次性把所有细节都塞进上下文里。

等大模型在运行过程中判断，某个 skill 真的需要调用了，它才会进一步利用读文件的能力，去读取这个 SKILL.md 里的主体内容，也就是 body 部分。

如果这个 skill 稍微复杂一点，读完 SKILL.md 其实还不够。因为下面往往还有 references 和 scripts。真正开始做事的时候，光知道原则、流程不行，还得继续看资料，还得继续执行动作。

所以你会发现，一个 skill 本质上是分层推进的。

第一层，是 name 和 description。
第二层，是 SKILL.md 的主体内容。
第三层，才是 references、脚本和其他补充信息。

三层一般就够了。如果再往下还很复杂，那通常就说明这个 skill 设计得太重了，应该继续拆，而不是把所有东西都塞在一个 skill 里。

那 skill 下面这些 references 和脚本，到底又是怎么执行的呢？

这里就能看到工具协同了。

skill 这个工具本身，其实不负责把事情做完，它的核心作用是给信息，告诉大模型现在有什么能力、这些能力大概是干什么的。真正读文档，靠的是 read。真正执行脚本、查看目录、搜索文件、调用命令，靠的是 bash。

也就是说，skill 先把能力暴露给大模型，让它知道现在有什么可以用。等它判断需要进一步理解某个技能时，再通过 read 去读取 SKILL.md 的主体内容。如果里面还有 references，就继续用 read 往下读。如果接下来需要执行脚本或者做具体动作，那再交给 bash。

看到这里，其实重点已经不是某一个工具本身有多强了，而是这些工具能不能形成顺畅的接力。

skill 不负责执行，read 不负责推理，bash 也不负责理解业务。每个工具只做自己最擅长的一部分，最后再由大模型把这些信息和动作串起来，形成完整的执行链路。

所以在设计工具的时候，不能把一个工具孤立开来看，也不要总想着做一个大而全的万能工具。更重要的是，要结合大模型和其他工具一起看，去设计它们之间的分工关系、信息流动和调用时机。

核心要想清楚，什么时候该给大模型什么信息，这些信息又会怎样影响它下一步的判断和决策。

其实这理解这就是所谓的 context engineering。

它的重点不是单纯往上下文里塞更多东西，而是在合适的时机，把合适的信息放进大模型的窗口里，让它在一个更合理的环境里推理。

那它和之前的 prompt engineering 到底差别在哪？

表面上看，当然有两层变化。

一层是信息组织方式变了。以前更像是把一整块信息提前准备好，现在则是把信息拆开，分散在 skill、文档、脚本和工具里，按需调用。
另一层是信息加载方式变了。以前更偏静态加载，现在更偏动态加载。

但我觉得，这些都还不是最本质的。

最本质的变化，其实是人和大模型的协作方式变了。

prompt engineering 的核心，还是人在告诉 AI 应该怎么做，推理过程本质上还是人主导的。

但 context engineering 不一样，它更像是人和工具一起给信息，再由大模型自己基于这些信息去推导下一步该怎么做。

一个是人主导流程。
一个是大模型主导推理。

我觉得这才是两者最本质的区别。
