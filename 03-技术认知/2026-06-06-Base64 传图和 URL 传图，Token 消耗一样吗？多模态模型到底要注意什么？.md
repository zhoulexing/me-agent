对于一个多模态模型来说，通过 Base64 上传一张图片，和通过 URL 上传一张图片，背后消耗的 token 是一样的吗？另外我们在使用多模态模型时需要注意哪些事项？

今天就以 Claude Code 和它背后的 Claude 模型为例，聊一下这两个问题，其他的模型所需要关注的点基本大差不差。

首先看第一个问题。

如果一张图片转成 Base64，你会看到它变成一大串字符，而且内容非常长，里面全是看不懂的字母、数字和符号。

但如果这张图片是一个 URL，可能也就几十个字符。

所以我们很自然会以为，这两种方式对模型的 token 消耗不一样。比如 RUL 就几十个 token。Base64 就达到几千上万的 token。

这个想法很好理解，但结论其实是不对的。

因为对于多模态模型来说，一张图片最终怎么计费、怎么进入模型，不取决于你是用 Base64 传过去，还是用 URL 传过去。

真正影响视觉 token 的，是图片本身被模型处理之后产生的视觉 token。

更具体一点说，它跟图片的尺寸、像素数量、模型的视觉编码方式有关，而不是跟你前面用什么形式把图片交给 API 有关。

所以在实际调用模型的时候，如果你的目标只是让模型读图，就不用纠结到底是上传 CDN 变成 URL，还是直接转成 Base64。

哪个方便就用哪个。URL 的优势主要是请求体更小、传输更清爽、复用更方便。

Base64 的优势是本地文件可以直接塞进请求里，不需要再额外上传到服务。

好，理解了第一个问题之后，接下来我们看第二个问题，就是在使用多模态模型是要注意的两个限制。

第一个是传输层面的限制，也就是 API 层面的限制。

比如这张图片最终放进请求里，大小不能超过多少。

第二类是视觉处理层面的限制。

比如模型真正看图时，图片长边最大能支持多少像素，超过之后会不会被缩放。

以 Claude 的模型为例，API层面单张图片大小好像是 5M，具体还是要以官方的文档为准。

所以我们在处理AI相关的工程时，需要针对图片做一些特殊的处理，就是最终进入模型侧，不能超过他的限制。

这里其实还有个坑在里面，就是一张图片在转成 Base64 后，体积会膨胀33%，这背后是因为图片文件背后的3个字节，转成 Base64 后会变成 4 个字符。所以如果 API 限制是 5M, 还是以 Base64 的方式传输的话，就需要把图片的大小控制在 3.75M。

但这里要注意，不是说有了这些限制之后，我们就把问题甩给用户，让用户自己去压缩图片、自己去判断大小、自己去处理格式。

接下里我们看下claude code是如何来处理的？我们以一张大的 PNG 图来举例。

首选它会先尝试做一次 PNG 压缩。PNG 压缩的好处是无损。

压完之后，再判断是否已经进入限制范围。

如果还不行，就进入第二步。

它会尝试把 PNG 转成 JPEG。

因为很多截图、照片类图片，用 JPEG 会比 PNG 小很多。

但 JPEG 是有损压缩，所以这里通常会按照不同质量等级逐步尝试。

比如先用 80 的质量压一遍。

如果还是太大，再尝试 60。

还不行，再尝试 40、20这样。

这个过程其实很像一层一层往下试，优先保证质量，实在不行再牺牲一点清晰度。

如果光转 JPEG 还不够，那就进入第三步，缩小图片尺寸。

因为很多时候，一张图真正让模型理解，并不需要保留完整的超大分辨率。

尤其是截图、网页、界面类图片，如果原图特别大，先按比例缩小，反而更符合模型实际可处理的范围。

缩小之后，它可能还会再做一次 PNG 压缩。

如果还是超出范围，就再转成 JPEG，再按照 80、60、40、20 这些质量梯度继续压。

最后如果所有办法都试过了，图片还是没办法压到限制以内，那就只能抛出错误。

这就是它的兜底逻辑。

先尽量无损。

不行就转格式。

再不行就降低质量。

还不行就缩小尺寸。

最后才报错。

这是第一个限制，接下来我们再看第二个限制。

也就是多模态模型在看图时，对图片也是有要求的，比如claude 的 Opus4.6, 图片的最长边要控制在 1568 像素以内，但到了 4.8 之后，这个限制增加到了 2576 像素。

除此之外对格式也有要求，比如 claude 模型主流的 PNG、JPEG、WEBP、GIF图片格式能支持，但其他的就不支持了。

所以针对这个限制，我们也需要对图片做处理，让图片进入模型前最长边控制在对应模型的限制范围内，且格式要转换成模型侧能够支持的。

最后总结一下。

第一，Base64 和 URL 只是图片传输方式，不是视觉 token 的本质来源。一张图片真正消耗多少视觉 token，主要取决于图片尺寸和模型的视觉处理方式。另外一张图转成 Base64 后会有体积膨胀的问题。

第二，在用户和模型之间需要做图片预处理，包括压缩、转格式、降质量、缩放和兜底报错。

所以我们在做智能体的时候，不能只盯着模型本身有多强。模型前面那一整套工程处理同样重要：文件怎么接收，图片怎么压缩，格式怎么转换，异常怎么兜底，都会直接影响最终效果。很多时候，用户感受到的智能，并不只是模型能力，而是模型能力加上工程系统一起配合出来的结果。
