今天是2026-04-06号。

最近Authropic宣布，Claude Pro和Max订阅用户，不得将订阅额度用于OpenClaw等第三方Agent框架，如果要想继续使用，就得切换成按量付费的API。这让很多用户都懵逼了，因为订阅制其实不用关心背后消耗了多少token，每个月是固定的成本，可以放心大胆的去使用，但按量付费不一样，它是不可预期的，每一次的调用，每一轮的Agent的循环，都是在持续计费。对于长程任务，有可能一晚上会把你一个月的工资给搭进去。

刚开始看到这个消息，我以为Authropic是为了限制竞争对手，毕竟OpenClaw这类Agent框架，一旦用得好，很容易对模型厂商自身的产品形态形成冲击，而且OpenClaw的创始人Peter还加入了OpenAI。

然后今天看到一篇公众号文章，主要介绍了小米 MiMo大模型负责人罗福莉对这件事的看法，我觉得从她说的这个角度来看，Authropic好像又显得没那么讨厌了。

首先Authropic官方对这件事给出的解释是，订阅制的定价模型原本是基于个人用户正常强度使用而设计的，但OpenClaw这类自动化代理工具的使用强度远远超出了预期，重度用户每月200美刀的订阅费用，缺消耗了价值5000美刀的算力资源，给Authropic带来了巨大的成本压力。

罗福莉对于Claude订阅制的理解是： 一套设计精良的算力均衡分配系统。它大概率是不赚钱的，甚至是亏钱。而且她也看了OpenClaw的上下文管理，给的结论是很差，一个用户请求之内，会触发多轮低价值的工具调用，每次都作为独立API请求发出，每次都携带一个很长的上下文窗口（往往超过 10 万 token）。折算下来，每个用户请求实际触发的 API 调用次数，是 Claude Code 自身框架的好几倍。换算成 API 定价，真实成本大概是订阅价格的几十倍。

所以她认为：对于Authropic将这类Agent框架进行订阅制调用的封禁，长期来看是有价值的，会倒逼工程进步。推动这些框架认真改进上下文管理、最大化Prompt缓存命中率复用以处理的上下文、消减无效token消耗的算力。痛苦终将转化为工程纪律。

作为一个工作很多年的程序员，说实话，我是有点工程纪律洁癖的，所以对于她的说法我也是认同的。而且现在很多的代码都是基于AI来生成的，生成完之后只要能跑就行，短期看效率是上来了，但长期来看是在不断积累技术债。当然我也不是反对AI写代码，因为我现在也差不多一年不手写代码了，只不过对于AI生成的代码需要定期的去清理技术债务。
