最近有两个开源项目挺火的，一个叫 OpenCLI，一个叫 CLI-Anything。那它们之间到底有什么区别？以及在特定的业务场景中该怎么选？

正好有一个网友也让我说一下它们，所以就来了。

先说 OpenCLI，它的核心是把网页、浏览器和一部分 Chromium 应用，封装成一套 Agent 更容易调用也更友好的 CLI，最终也是为了实现浏览器的自动化。

我之前写过一篇内容，叫《一文彻底了解浏览器自动化》，详细介绍了cdp、playwright、browser-user、midscene、browsermcp之间的联系和区别，没看的可以先回头了解下。

那今天所说的 OpenCLI 是怎么做的呢？

可以拆成两层来看。第一层，是浏览器这一侧。

OpenCLI 会在浏览器里安装一个扩展程序。这个扩展程序可以理解成浏览器里的一个代理程序，因为它运行在浏览器环境里，所以能接触到当前页面、标签页、DOM、网络请求、页面状态，也能进一步调用浏览器底层暴露出来的控制能力。

这里的底层控制能力，核心还是 CDP，全称是Chrome DevTools Protocol，这是浏览器自动化所绕不开的。

第二层，是本地这一侧。

你下载 OpenCLI 之后，本地会有一个命令行程序，同时还会有一个桥接服务。你或者 Agent 执行命令时，比如打开某个页面、点击某个按钮、提取某段内容，这个命令不会直接作用在网页上，而是先进入本地 CLI。

本地 CLI 会先解析这个命令到底要做什么。比如你想要的是打开页面，还是点击元素。解析之后，本地桥接服务会把这个操作请求发送给浏览器扩展程序，然后由扩展程序来完成实际的执行。

所以整个链路可以理解成：Agent 发出一个命令，本地 CLI 接收并解析这个命令，然后通过桥接服务把请求传给浏览器扩展。浏览器扩展进入具体页面环境，借助 CDP 这类底层能力完成点击、输入、提取、截图、监听网络等动作。最后，操作结果再沿着这条链路返回给本地 CLI，并最终返回给 Agent。

OpenCLI 里面也内置了很多网站和应用的适配器，当然你也可以基于它提供的能力来把你的网站变成一个本地的CLI。

那它和 browser-use、browsermcp 有什么区别？

browser-use是基于Playwright 这类浏览器自动化框架，当然也渐渐的往更底层的 CDP 方向走，而且需要有一个Chromeium内核，而OpenCLI不需要，只需要装个浏览器扩展程序就好。

browsermcp 是官方提供的，也是需要安装一个插件，但是是通过 MCP来通讯的，把浏览器能力暴露给 AI 应用。

再说 CLI-Anything，它 和 OpenCLI 的出发点完全不一样。OpenCLI 的入口是网页和浏览器。CLI-Anything 的入口是已有程序、代码库和 API。

它解决的问题是，很多软件本身已经很强了，但并不是为 AI Agent 准备的。有的软件有 GUI，但 Agent 不适合一直去点界面。有的软件有 API，但接口太底层、太碎、参数太多，Agent 每次都要重新理解。

比如一个图像编辑工具，一个数据分析工具，一个内部业务系统。这些工具本身都有程序或API文档，对人来说，可以慢慢看文档、调接口。但对 Agent 来说，这个成本太高了。

这就是 CLI-Anything 想解决的问题。它的核心思路是，根据原有程序、代码结构或者 API 文档，生成一套标准 CLI，再配套生成 Skills。

它生成出来的结构大概会分成几层。有一个入口文件，负责命令注册和执行。有一个 core 目录，负责封装 API、认证和底层调用。有一个 commands 目录，负责定义具体命令，比如订单相关命令、用户相关命令、项目相关命令。还有一个 skills 目录，负责告诉 AI 这些命令怎么用，参数怎么传，返回结果怎么看，失败了应该怎么办。

所以 CLI-Anything 的本质，可以概括成一句话：它不是重新做一个软件，而是基于原有程序做一层封装，让这个软件对 AI 更友好。原来的软件可能是给人点的，也可能是给程序员调接口的。但经过 CLI-Anything 这一层之后，它就变成了一套更适合 Agent 理解、调用、组合和验证的命令系统。

所以 OpenCLI 和 CLI-Anything 的区别，其实一句话就可以说清楚。

OpenCLI 是从网页和浏览器切入，把网页工作流变成 CLI。CLI-Anything 是从软件、代码库和 API 切入，把已有软件能力变成 CLI。

再来看下为什么在AI时代一定要把网页的操作或者软件程序变成CLI和SKILL？

因为不管是网页操作，还是软件程序，只要想让 Agent 稳定使用，都需要一层更适合 AI 的入口。

先看 GUI。GUI 是给人设计的。人可以看页面、找按钮、理解布局、判断状态，所以页面做得越直观，人用起来越舒服。但对 AI 来说，GUI 不是最稳定的入口。

因为 GUI 依赖视觉，页面一变，AI 就要重新理解。按钮换了位置，弹窗多了一层，字段名字改了一下，DOM 结构调整了一点，Agent 都可能误判。它不是不能操作 GUI，而是每次操作都要重新观察、重新推理、重新确认，稳定性很难保证，而且成本也很高。

再看 API。API 看起来更适合机器调用，但它也不是天然适合 Agent。因为 API 往往太底层、太分散、太工程化。一个真实任务，可能要调好几个接口，要处理认证、分页、字段、错误码，还要理解接口之间的调用顺序。

API 更像一堆零件，告诉你系统有哪些能力；但 Agent 需要的不是零件清单，而是可以直接执行的任务动作。

只有 CLI 还不够，因为 Agent 即使知道有这个命令，也不一定知道什么时候用、怎么用、用完怎么看结果、失败后怎么处理。

这时候就需要 Skill，是写给 Agent 的操作说明书。它告诉 Agent：有哪些命令，分别适合什么场景，参数怎么传，执行顺序是什么，返回结果怎么看，哪些操作有风险，失败之后怎么恢复。

所以 CLI 和 Skill 是一组配套关系。

最后，AI 时代的软件设计，不仅要考虑对人友好，还要考虑对AI友好。这也是两个工具能够火起来的原因。
