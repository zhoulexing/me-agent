今天是2026年3月31号，最近经常听到 Harness Engineering，那它到底是什么意思？有哪些组成部分？又为什么会这么火？

接下来我们先通过几个例子来感受一下。

第一个例子，是一个关于编辑的工具。我们知道，大模型其实是没有动手能力的。要让它真正干活，就得给它提供各种各样的工具，其中一个很关键的工具，就是编辑工具。

但问题是，业界主流智能体的编辑工具，成功率其实都不高。然后有一个人叫 Can Duruk，他独立维护了一个项目，专门研究这件事。

他发现，以前很多编辑方式都有一个共同的问题，就是太依赖模型去“精准复现”原文。比如有的方案要求模型输出特定格式的 diff，有的方案要求模型把原文一字不差地抄回来再替换。听起来好像合理，但实际上，只要模型少一个空格、多一个换行，或者记错一小段文本，整个编辑就可能失败。

后来他做了一个叫 Hashline 的方案。核心思路很简单，就是模型读文件的时候，不再只看纯文本，而是给每一行都加一个很短的哈希标签。这样模型在编辑的时候，不需要费劲去复述原文，只要引用对应的标签，就能定位到该改哪一行。

结果非常夸张。有些模型原来改代码的成功率特别低，上了这套方案之后，成功率直接翻了很多倍，甚至有的是十倍级别的提升。

这个例子很说明问题。很多人以为，模型表现不好，是模型不够聪明。但实际上，很多时候不是模型不行，而是你给它的“手”和“操作方式”不行。也就是说，问题不一定出在大脑，有可能出在工具。

第二个例子，是有人用 AI Agent 在52天里写了35万行生产代码。

这件事听起来很猛，但他后来总结出了一个特别重要的问题，就是技术债在 Agent 时代会被放大得非常快。

什么意思呢？比如你今天为了赶进度，写了一个临时补丁；或者你懒得走正常架构，直接绕过 Service 层查数据库；又或者你写了一个硬编码的魔法数字。人类工程师看到这些东西，很多时候心里是知道的，知道这只是临时方案，知道这东西不优雅，甚至知道以后最好别这么写。

但 Agent 不是这样。它看到代码库里已经有这种写法，就会默认这是一种被允许的范式。下次再遇到类似问题，它就会继续这么写。再下一次，它还会继续复制。慢慢地，一个本来只是局部的脏写法，就会扩散成整个系统的坏味道。

所以在传统软件工程里，技术债可能只是“留了个坑”。但到了 Agent 时代，技术债更像是一种会自我复制的病毒。你不及时管住它，它就会越长越快。

第三个例子，是子 Agent。

有团队在做企业级 Agent 落地的时候发现了一个特别真实的问题，就是 Agent 的上下文会慢慢变脏。

它每读一次文件、每调一次工具、每跑一次搜索，都会往上下文窗口里塞很多东西。刚开始可能还好，但信息一多之后，它就会出现一种状态：明明是个不复杂的问题，它却开始跑偏、开始混乱、开始做出一些很奇怪的决策。

那怎么解决呢？有些人第一反应是，把上下文窗口做得更大。但后来他们发现，真正有效的办法不是一味加大窗口，而是做隔离。

具体做法就是，把一个大任务拆给多个子 Agent。父 Agent 负责规划，子 Agent 负责执行。每个子 Agent 都在自己的小上下文里处理一个局部问题，处理完之后，只把结果和少量关键信息返回给父 Agent。

这样做的好处是什么？就是父 Agent 不会被大量细碎过程污染，始终能保持比较清醒的状态。说白了，子 Agent 不只是为了分工，更像是一个“上下文防火墙”。

第四个例子，是反馈机制。

有团队一开始的做法是，Agent 每改完一次代码，就把整套测试全跑一遍。听起来很严谨，对吧？但问题是，测试一旦很多，通过日志就会特别长。几千行“全部通过”的输出，全都塞进上下文里，其实没什么价值，反而会干扰 Agent 后续的判断。

后来他们总结出一句我觉得特别有意思的话，叫“成功应该保持沉默，失败才应该发出声音”。

什么意思？就是格式化通过了、类型检查通过了、测试通过了，这些事情不用大张旗鼓告诉 Agent，因为这些信息对它下一步行动没有帮助。但只要失败了，就把错误信息清晰地甩给它，让它围绕错误去修。

这个思路背后其实也很重要。不是说你给 Agent 的反馈越多越好，而是你要给它“有用的反馈”。无效信息太多，本质上也是一种噪音。

说到这里，其实你会发现，这几个例子虽然表面上讲的是不同的事，但背后指向的是同一个问题：

不是单纯让模型变强，而是怎么给模型搭一个更适合它工作的环境。

这时候，Harness Engineering 这个词就出来了。

你可以把 Harness 理解成缰绳、马具、护具这一整套东西。马当然很重要，但如果没有这整套系统，马再强，也很难稳定地往你想要的方向跑。

所以 Harness Engineering，如果用我自己的理解去说，它不是在研究“怎么把提示词写得更花”，也不只是研究“给模型塞什么上下文”，而是在研究：

怎么把工具、上下文、约束、反馈、验证这些东西，组合成一套能让 Agent 稳定干活的运行系统。

换句话说，Harness Engineering 关注的，不是单次问答，而是一个 Agent 从开始接任务，到中间执行，再到最后交付，这整条链路应该怎么设计。

那它大概包含哪些部分呢？我觉得至少有下面这几个。

第一，是上下文管理。

你要决定，什么信息该给 Agent 看，什么信息不该给它看；什么时候该压缩上下文，什么时候该切子任务；以及怎么避免上下文越跑越乱。因为不是信息越多越好，而是信息越准越好。

第二，是工具系统。

Agent 不会凭空做事，它一定得依赖工具。比如读文件、改文件、搜索代码、执行命令、调用接口，这些都属于工具层。工具设计得好，Agent 就像拿到了合适的手脚；工具设计得差，它再聪明也会经常摔跤。

第三，是反馈回路。

Agent 做完事之后，你怎么验证？验证通过了要不要告诉它？失败了之后怎么把错误喂回去？这些都属于反馈回路。一个好的反馈回路，能让 Agent 快速修正；一个差的反馈回路，只会让它在噪音里打转。

第四，是规则和约束。

比如哪些目录不能动，哪些接口必须经过哪一层，哪些写法不允许出现，哪些检查必须通过之后才能提交。以前这些规则主要是约束人，现在这些规则同样也要约束 Agent。而且很多时候，越是明确的约束，对 Agent 反而越友好。

第五，是技术债治理。

因为 Agent 特别会“模仿现有代码库”。所以如果代码库本身不干净，那它就会不断复制这些坏模式。也就是说，你不只是要写新功能，还要持续清理旧问题，不然 Agent 会帮你把历史包袱越滚越大。

讲到这里，再顺手说一下它和 Prompt Engineering、Context Engineering 的区别。

Prompt Engineering 更像是在研究：我这句话到底该怎么说，模型才更容易听懂。

Context Engineering 更像是在研究：我到底该把哪些信息，在什么时机，以什么结构给到模型。

而 Harness Engineering 再往前走了一步。它关注的是：这个 Agent 工作的整个环境，该怎么搭。

所以如果简单粗暴一点理解的话：

Prompt Engineering 解决的是“怎么说”。

Context Engineering 解决的是“给它看什么”。

Harness Engineering 解决的是“整个系统怎么跑”。

也正因为这样，我觉得 Harness Engineering 之所以最近越来越火，本质上是因为大家已经慢慢发现，模型能力本身不再是唯一瓶颈了。

以前模型不够强，大家会把注意力放在模型本身，或者放在提示词上。但现在模型越来越强、Agent 也越来越便宜之后，新的瓶颈就开始转移了。

转移到哪里？转移到你到底会不会搭系统。

同样一个模型，给不同的人、放到不同的运行环境里，最后跑出来的结果可能完全不一样。有人用起来像一个很能打的工程师，有人用起来像一个到处闯祸的实习生。差别往往不只是模型参数，而是背后的 harness。

所以Harness Engineering 不是一个特别花哨的新概念，它更像是 AI Agent 时代的软件工程基本功。

你可以把它理解成，过去我们在管理代码、管理系统、管理工程复杂度；现在，只不过是把“管理对象”进一步扩展到了 Agent 本身。
