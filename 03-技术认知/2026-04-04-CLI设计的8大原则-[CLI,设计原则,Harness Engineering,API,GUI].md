# CLI设计的8大原则

## 背景

最近飞书、钉钉、企业微信都陆续推出了自己的命令行工具以及配套的技能包，可以更方便的结合各种智能体来操作背后的应用。

不仅如此，还有像开源的 AnythingCLI、OpenCLI 这类开源工具能非常快速的把一个网页或者一个系统，直接变成命令行工具来使用。

## CLI与API、GUI、SKILL的关系

在 CLI 设计之前，先理解 CLI 和 API、GUI、SKILL 之间的关系，以及有了 API，为什么还要搞一层 CLI？

### 场景：新人入职操作交给 AI 智能体

**方案一：RPA（依赖 GUI）**
- 模拟人工操作：打开网址 → 进入页面 → 点击按钮 → 输入信息 → 保存
- 缺点：需要 AI 学会看图，稳定性差，成本高
- 本质：GUI 服务于人，对 AI 不友好

**方案二：直接调用 API**
- 需要处理多个 API 之间的依赖关系
- 需要把依赖信息给到 AI
- 问题：复杂度高，容易出错

**方案三：封装成 CLI**
- 在 CLI 层面封装逻辑
- 错误处理和命名更友好
- 不动原有系统 API
- Token 消耗更少
- 配合 SKILL 提供使用说明

## CLI的三大核心要素

1. **要处理的对象**
2. **处理的动作**
3. **对应的参数**

所有设计都是围绕这三点展开。

## 8大设计原则

### 原则一：命令排布遵循名词在前，动词在后

**示例：**
- ✅ 推荐：`cli user search`（对象在前，动作在后）
- ❌ 避免：`cli search user`（铺平写法，动词在前）

**优势：**
- 可搭建命令目录树
- 方便扩展（user、calendar 等模块）
- 每个叶子节点都可通过 help 查看操作

### 原则二：长参数优先，杜绝简写

**原因：**
- 命令行对大小写敏感
- `-v` 和 `-V` 含义完全不同
- 简写容易让 AI 理解出错

**示例：**
- ✅ 推荐：`--name`
- ❌ 避免：`-n`

### 原则三：输出格式要方便后续处理

**推荐格式：JSON**

**优势：**
- 与管道操作适配度高
- 可用 jq 等 JSON 处理器快速提取字段
- 示例：`cli user list | jq .total`

### 原则四：干跑（dry-run）优先

**适用场景：** 创建、修改等有副作用的命令

**实现方式：** 添加 `--dry-run` 参数

**作用：**
- 预览执行结果和后果
- 确认无误后再真正执行
- 最大程度减少误操作风险

### 原则五：精细化设计退出码

**传统做法：** 只用 0（成功）和 1（错误）

**精细化设计：**
- `0` - 执行成功
- `1` - 普通业务错误
- `2` - 参数错误
- `3` - 权限不足
- ...（根据业务定义更多）

**价值：** 让 AI 精准判断错误原因，自主做出对应处理

### 原则六：遵循幂等设计，优先声明式

**命令式 vs 声明式：**

| 类型 | 命令 | 问题 |
|-----|------|-----|
| 命令式 | `cli user create` | 重复执行可能出错或多创建 |
| 声明式 | `cli user ensure` | 存在就不操作，不存在才创建 |

**或者使用参数方式：**
```bash
cli user create --if-not-exists
```

**价值：** 无论执行多少次都不会出问题，安全性更高

### 原则七：错误提示友好，给出明确下一步操作

**不能只简单报错，要：**
1. 明确说明缺少什么具体权限
2. 告诉通过什么方式、什么命令能获取权限
3. 给出完整的下一步行动方案

**示例：**
```
错误：缺少 calendar:write 权限
解决方式：运行 cli auth add --scope calendar:write 获取权限
```

### 原则八：配备按需级别的帮助文档（最重要）

**要求：**
- 完善的 `--help` 帮助文档
- 支持层级化、按需查看

**示例：**
- `user --help` - 查看用户模块的操作
- `user create --help` - 查看创建用户的具体参数

**价值：** 需要什么内容就查对应的层级，精准又高效

---

## 总结

以上就是 CLI 设计的 8 大核心原则：

1. 名词在前，动词在后
2. 长参数优先，杜绝简写
3. 输出格式方便后续处理（优先 JSON）
4. 干跑（dry-run）优先
5. 精细化设计退出码
6. 遵循幂等设计，优先声明式
7. 错误提示友好，给出明确下一步操作
8. **配备按需级别的帮助文档（最重要）**
