今天是2026年4月2号。昨天我聊了CLI和SKILL在规模化场景下，如何处理超大数据量的问题。后面我就想到，智能体工具基本都会针对这种大数据量做专门处理，不然早就扛不住了。于是我就很好奇，在这种场景下，一般的bash命令是怎么处理的？

正好前几天 Claude Code 的源码泄露了，我也下载了一份，可以借此看看它针对这类大内容场景是如何处理的。

发现它其实也分了好几档处理逻辑：在执行命令时，如果命令返回了输出内容，它会先将其写入临时文件。

当然这个写入也不是无限制的，一旦内容超过 5G，就会停止写入，并提示内容体积过大。

那停止写入之后会怎么处理呢？它会从这个临时文件里，只截取前 64MB 的内容，拷贝到 tool results 目录下。

把这 64MB 的文件拷贝到目录后，它也不会把完整内容返回给上下文，依旧只截取其中 30KB 返回到上下文里。后续如果需要继续分析，也只能基于这 64MB 的文件内容来进行。

说到这里，大家肯定会想到一个问题：这里的临时文件和 tool results 目录里的文件，该怎么处理？总不可能一直留在那儿占空间吧。

那这里就分几种情况了。如果输出小于 30KB，因为内容能直接读完，它会直接删掉临时文件，也不会存到 tool results 目录里。

如果大于 30KB 但小于 64MB，它会把临时文件里的内容拷贝到 tool results 目录下，这时候就会同时存在两份文件。

同样，超过 64MB 的内容也是同理，它只会在 tool results 里保留 64MB 用于后续工作。

至于临时目录里那些超过 64MB 的原始内容，可能还会留在原地，但 tool results 目录下的文件，是有清理机制的。

下次启动 Claude Code 时，它会在空闲时自动检查：比如 tool results 里的文件如果存放超过 30 天，就会被自动清理。但如果你之后再也不打开 Claude Code，那 session 目录下的这些文件，可能就一直留在那里不会被清理了。

好，这就是今天 Claude Code 关于Bash 命令针对大内容的截断与处理逻辑。
