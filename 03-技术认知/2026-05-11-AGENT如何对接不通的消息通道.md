# AGENT如何对接不通的消息通道

我们在使用飞书、钉钉、企业微信来对接 OpenClaw 或 Hermes 这类框架时，我不知道大家有没有想过一个问题：我们在使用过程中，发送给这些通道的不仅仅是文字，也可能是一张图、一个视频或者是一个PDF文件，这些文件或者视频最终是上传到这些通道消息侧的服务器，而我们Agent是部署在另外一套服务器上的，那这个agent到底是怎么处理消息通道侧服务器上的文件呢？

反过来，如果 Agent 自己生成了一份文件，比如一张图片、一份 CSV 表格、一个分析报告，它又是怎么通过飞书、钉钉、企微发出来的？

接下来分三个部分来分享下关于这块的处理。

## 第一部分：消息接入，需要先做统一抽象

不同消息通道的格式是不一样的。飞书有飞书的消息结构，钉钉有钉钉的消息结构，企业微信也有自己的消息结构。如果 Agent 直接去适配每个平台，就会导致 Agent 层混入大量平台细节，后面接入新的通道也会越来越麻烦。

所以更合理的设计，是在消息通道和 Agent 之间加一层消息抽象层。不管消息来自飞书、钉钉还是企微，进入 Agent 之前，都先转换成统一的消息格式。这样 Agent 层只需要关心：这是谁发来的消息，属于哪个会话，消息类型是什么，是文本、图片、视频，还是文件。

这里最容易被忽略的是文件处理。用户在飞书里发了一个 PDF，这个文件一开始并不在你的服务器上，而是在飞书自己的文件服务里。消息里可能只有一个 file_key 或者 media_id，如果直接把这个标识丢给 Agent，Agent 其实是读不到的。

所以消息接入层还要负责把文件下载下来。也就是说，要先从飞书、钉钉、企微这些平台的文件服务里，把图片、视频、PDF、CSV 等文件下载到 Agent 可以访问的位置，再把本地路径或可访问地址交给 Agent。只有这样，Agent 才能真正读取和处理这些文件。

## 第二部分：Agent 层要处理 session（会话）队列和并发

消息接进来之后，还不能直接无脑丢给 Agent。因为这些消息通道不会限制用户连续发消息，哪怕 Agent 正在处理上一条消息，用户仍然可以继续发第二条、第三条。对于聊天软件来说，这很正常；但对 Agent 来说，如果不做处理，上下文很容易乱。

比如用户先发“帮我总结这个文件”，接着又发“提炼成三点”，再发“语气口语一点”。这三句话其实属于同一个任务的连续补充，应该放在同一个 session 里按顺序处理。如果三条消息被并发交给 Agent，就可能出现前后顺序错乱，甚至不同任务互相干扰。

所以同一个 session 通常需要一个任务队列。前一条消息还没处理完，后面的消息可以先排队，等前一个任务结束之后再继续执行。这样可以保证同一个会话里的上下文是稳定的，Agent 处理起来也更符合用户的真实意图。

不过，不同 session 之间是可以并发的。比如张三在飞书里问问题，李四在钉钉里上传文件，王五在企微里让机器人生成报告，这些会话之间没有强依赖，可以同时处理。但并发也不能无限放开，如果一个机器人同时被很多人使用，又没有并发限制，服务器性能差一点就可能直接被打挂。

## 第三部分：消息接出，也需要把 Agent 输出转换成通道消息

Agent 处理完任务之后，会产生不同类型的输出。有的是助手回复，比如一段文字；有的是工具执行结果；有的是思考过程；还有的是文件，比如生成了一张图、一份表格或者一个报告。这些内容都需要通过消息通道再发回给用户。

普通文本相对简单。Agent 生成一段回复后，可以通过 hook 机制监听到这条消息，然后调用对应平台的接口，把它发回飞书、钉钉或企微。这样 Agent 层不用关心具体平台怎么发消息，只需要产出统一的消息内容。

但文件发送会复杂一点。因为 Agent 生成的文件，通常只是服务器上的一个本地文件，比如 `/tmp/result.csv`。这个路径对服务器有意义，但对飞书、钉钉、企微没有意义，机器人不能直接拿一个本地路径去发送文件。

所以消息接出层还需要专门处理文件上传。比较常见的做法是，当工具生成文件时，通过 hook 把文件路径记录下来；等发送消息时，通道层先把这个文件上传到飞书、钉钉、企微的文件服务，拿到平台返回的 file_key、media_id 或类似标识，再通过机器人把这个文件发出去。

也可以单独实现一个发送文件的工具，比如 `send_file` 或 `send_image`。Agent 只需要调用这个工具，并告诉它要发送哪个文件；工具背后负责把文件上传到消息通道服务，再把它发给用户。这样职责会更清楚：Agent 决定发什么，工具负责怎么发，消息通道负责对接平台。

## 最后

讲到这里，其实我更想强调一个变化：在 AI 时代，做软件开发，不一定要像以前一样，把每一个接口、每一个参数、每一段代码都提前研究得特别清楚。很多时候，你真正需要掌握的是原理层，也就是这套系统为什么要这样设计，中间有哪些关键环节，每个环节解决什么问题。

就拿 Agent 对接飞书、钉钉、企微这件事来说，如果你已经理解了它的设计机制，后面的实现其实就很简单了。你知道消息进来之后要先做统一抽象，知道文件要从消息平台下载下来，知道同一个 session 要排队处理，知道不同 session 可以并发，知道 Agent 生成的文件要先上传到消息平台，再通过机器人发出去。只要这些关键点想清楚了，剩下的很多代码实现，就可以交给 AI 来完成。
