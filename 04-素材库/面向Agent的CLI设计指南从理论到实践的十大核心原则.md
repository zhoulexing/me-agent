# 面向Agent的CLI设计指南：从理论到实践的十大核心原则

## 🤖 Agent时代的CLI变革（背景）

**行业趋势**
- **工具使用主体转变**：Agent正成为软件的主要用户，其交互方式从GUI转向CLI（命令行界面）。
- **传统设计不适用**：传统CLI规范面向人类，而Agent具有不同的认知特点和失败模式，需针对性设计。
- **核心挑战**：如何让CLI同时满足Agent的准确性需求与人类的使用习惯，成为产品CLI化的关键问题。

---

## 🧠 Agent的认知特点与弱点（基础）

### (一) Agent的命令行直觉边界

- **训练数据基础**：LLM训练数据包含约5% GitHub代码和2% Stack Overflow问答，对命令行有天然「直觉」。
- **能力局限**：BFCL V4显示，最强模型在复杂多步工具调用中准确率不足100%；SWE-bench数据显示前沿模型仅能解决约70%编码任务。
- **行为模式稳定**：即使模型能力提升，Agent仍通过`--help`、解析输出、拼命令、看退出码的流程使用CLI。

### (二) Agent的天然弱点

| 弱点类型 | 具体表现 | 典型案例 |
| :------- | :------- | :------- |
| **大小写敏感短参数** | 同一字母大小写含义完全不同，增加错误概率 | `grep -i`（忽略大小写）与`grep -I`（忽略二进制文件） |
| **交互式提示** | 无法处理`[y/N]`等交互式问答 | AWS CLI v2默认启用`less`分页导致CI任务挂起 |
| **幻觉参数** | 自信使用不存在的参数，开源模型尤其严重 | 模型虚构不存在的类名并持续错误调用 |
| **非结构化输出** | 难以解析人类可读的表格，依赖结构化数据 | JSON输出可直接提取字段，表格需猜测解析 |

## 🎯 面向Agent的十大CLI设计原则（核心）

### 原则一：名词在前（noun-verb结构）

- **设计逻辑**：命令结构采用「资源-操作」顺序，如`docker container ls`，而非「操作-资源」（如`create-pr`）。
- **优势**：符合Agent的树搜索习惯，通过`--help`逐层发现资源与操作，降低猜测成本。
- **典范案例**：Docker命令体系（`container`/`image`/`volume`等名词+`ls`/`rm`/`create`等动词）。

### 原则二：长参数优先

- **核心要求**：所有参数必须提供长格式（`--verbose`），短格式（`-v`）仅作为人类便利选项。
- **关键价值**：
  - **语义自描述**：`--dry-run`比`-n`更清晰传达「预演」功能
  - **消除歧义**：`--version`与`--verbose`无大小写混淆风险
  - **LLM概率优势**：训练数据中长参数与语义绑定更强

### 原则三：输出是契约

- **标准规范**：
  - **stdout**：仅输出结构化数据（如JSON），作为API契约
  - **stderr**：输出人类可读的状态信息（进度、警告、日志）
- **版本控制**：结构化输出需保持向后兼容，新增字段安全，修改字段类型/名称视为破坏性变更。
- **实践案例**：飞书CLI支持JSON/NDJSON/table/CSV/pretty五种格式，GitHub CLI在管道传输时自动切换为tab分隔格式。

### 原则四：感知环境

- **自适应行为**：CLI需检测运行环境（终端/管道）并调整输出：
  - **终端（TTY）**：彩色输出、表格、进度条
  - **管道（pipe）**：纯文本、JSON、无颜色、无交互
- **最佳实践**：飞书CLI非TTY环境自动降级为JSON输出，钉钉CLI通过`--yes`参数跳过所有确认提示。

### 原则五：干跑优先（--dry-run）

- **核心功能**：所有产生副作用的命令必须支持`--dry-run`，提供零成本试错机制。
- **实现要求**：输出结构化JSON diff，明确展示创建/修改/删除的资源。
- **进阶设计**：Lightning Labs使用专门退出码（10）标识「干跑成功」，便于Agent区分执行状态。

### 原则六：退出码控制

- **细粒度定义**：需超越0（成功）和1（失败），提供更具体的状态编码：

| 退出码 | 含义 | Agent应对策略 |
| :----- | :--- | :------------ |
| 0 | 成功 | 继续执行后续命令 |
| 1 | 一般错误 | 读取stderr诊断 |
| 2 | 参数错误 | 修正参数重试 |
| 3 | 资源不存在 | 跳过或创建资源 |
| 4 | 权限不足 | 提示用户授权 |
| 5 | 冲突/已存在 | 跳过或更新资源 |

- **稳定性要求**：退出码语义需跨版本保持稳定，视为API契约一部分。

### 原则七：防住幻觉

- **输入验证**：严格校验URL（拒绝`javascript:`协议）、域名（过滤shell元字符）、输出路径（保护敏感目录）。
- **枚举约束**：参数优先使用枚举值（如`--format json|table|csv`），减少自由文本输入。
- **schema自省**：提供命令元数据查询，如`mytool schema user create`输出参数定义，飞书CLI已实现该功能。

### 原则八：幂等设计

- **声明式优先**：使用`ensure`或`--if-not-exists`等机制，确保重复调用结果一致。
- **典型案例**：
  - 命令式（风险）：`mytool user create --name "john"`（重复执行报错）
  - 声明式（安全）：`mytool user ensure --name "john"`（重复执行无副作用）
- **扩展实践**：飞书CLI通过`--idempotency-key`参数实现请求幂等性。

### 原则九：错误即指南

- **错误信息四要素**：
  - **错误类型**（机器可读，如`permission_denied`）
  - **描述**（具体原因，如「缺少calendar:read权限」）
  - **修复建议**（可执行命令，如`lark-cli auth login --domain calendar`）
  - **可重试标志**（布尔值，区分网络超时与权限不足）

### 原则十：帮助即大脑

- **高质量`--help`标准**：
  - 以2-3个使用示例开头
  - 明确标注必需/可选参数
  - 参数描述包含值域（如`--format json|table|csv`）
  - 控制在50行以内，避免信息过载

## 📊 主流CLI产品评估（实践）

### 飞书CLI

- **优势**：noun-verb结构清晰，三层架构支持按需抽象，`schema`命令提供API自省，五种输出格式，权限不足自动提示修复方案。
- **改进空间**：非TTY环境未自动切换JSON，退出码语义未文档化，`--idempotency-key`应用范围有限。

### 钉钉CLI

- **优势**：`--yes`参数语义自描述（AI Agent模式），`--mock`方便调试，帮助文本全中文（适配中文LLM），批量熔断防失控。
- **改进空间**：命令仅两层缺少快捷层，无schema自省，输出格式仅三种，帮助文本缺少示例。

## ✅ 开发清单（工具）

1. [ ] 命令结构为noun-verb层级，支持树搜索
2. [ ] 所有参数提供长格式，短格式可选
3. [ ] `--json`输出到stdout，状态信息走stderr
4. [ ] 检测TTY状态并自动调整输出
5. [ ] 副作用操作支持`--dry-run`并输出结构化diff
6. [ ] 退出码有文档化的细粒度语义
7. [ ] 参数使用枚举约束值域，输入严格验证
8. [ ] 支持schema自省命令
9. [ ] 关键操作设计为幂等或提供`--if-not-exists`
10. [ ] 错误信息包含类型、描述、建议和可重试标志
11. [ ] `--help`以示例开头，标注必需/可选，控制在50行内
12. [ ] 提供`--yes`/`--no-interactive`参数跳过交互

- **SWE-bench**：软件工程师基准测试，衡量模型解决真实编码任务的能力。
- **12 Factor CLI Apps**：CLI应用开发的12项原则，强调可移植性和稳定性。
- **Agent-CLI设计轴**：Lightning Labs提出的10个设计维度，涵盖幂等性、错误处理等关键要素。
- **Token效率与准确率权衡**：短参数（如`grep -irl`）省Token但高风险，长参数（如`grep --ignore-case --recursive`）虽多Token但语义明确，对Agent而言准确率优先于Token效率。
