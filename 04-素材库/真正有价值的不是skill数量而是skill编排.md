# 真正有价值的不是 skill 数量，而是 skill 编排

> 标签：#AI链接笔记 #Skill编排 #Agent工作流 #系统解释型
> 创建时间：2026-04-17 08:42:00

---

你有没有发现一个很真实的现象。

现在大家都在装 skill。

写作 skill，调试 skill，review skill，测试 skill，发布 skill，浏览器 skill，数据库 skill。装的时候很兴奋，感觉 Agent 马上就要从实习生变成专家组了。结果真到复杂任务里，问题还是那几个。该触发的时候没触发，不该触发的时候乱触发。前一个 skill 刚做完，后一个 skill 根本接不上。明明每个 skill 单看都挺对，串起来却像一屋子很厉害但互相不认识的人。

所以我最近越来越觉得，真正有价值的不是 skill 数量，而是 skill 编排。

因为 skill 解决的是你会不会。
编排解决的是你怎么一起做。

## 是什么

先把这个词讲清楚。

我理解的 skill 编排，不是把很多 skill 收集到一个目录里，也不是给 Agent 塞更多 SKILL.md 文件，更不是做一个 skill marketplace 就算结束。

skill 编排真正做的，是定义四件事。

第一件，什么时候触发哪个 skill。

第二件，多个 skill 之间是什么关系，是互斥、并行、串行，还是前后依赖。

第三件，谁来决定下一步该走哪个 skill，是主 Agent 判断，还是一个专门的调度层判断。

第四件，每个 skill 执行完之后，结果怎么被验证、沉淀、交给下一个 skill。

如果没有这四件事，skill 再多，也只是零散能力包。

就像你有一堆函数，不等于你有一个程序。
你有一堆服务，不等于你有一个系统。
你有一堆 skill，也不等于你有一个能稳定交付复杂任务的 Agent。

所以 skill 是能力单元。
编排是执行系统。

这个区别特别重要。

因为 skill 的本质是局部优化。
它负责把某一个局部任务做得更稳。
比如写 spec，更像 spec。
跑性能优化，更像性能优化。
做 code review，更像 code review。

但真实任务不是单点问题。
真实任务是链路问题。

一个复杂任务里，最难的部分往往不是某一步做不到，而是多步之间怎么接。
什么时候该研究，什么时候该规划，什么时候该执行，什么时候该停下来验证，什么时候该切换角色，什么时候该把结果回写成新的上下文。

这些都不是单个 skill 能解决的。
这些属于编排层。

## 为什么

为什么我会觉得，skill 编排比 skill 数量更关键。

因为一旦任务复杂起来，系统性能的瓶颈就不在单点能力，而在协同损耗。

你可以把这件事理解成三个层次。

第一个层次，是触发错误。

很多人现在做 skill，还是停留在看到关键词就触发。
用户说性能，就调 performance skill。
用户说测试，就调 test skill。
用户说上线，就调 deploy skill。

这在简单任务里还能凑合。
但复杂任务里会出大问题。

因为真实任务的意图，不是按关键词分布的。
它经常是混合的。

比如你让 Agent 优化一个页面加载速度。
这里面可能既需要 performance skill，也需要浏览器调试 skill，还需要 code review skill，最后可能还要补一个验证流程。
如果系统只是按关键词抓一个 skill，它很容易把优化性能理解成直接给几个建议，而不是进入完整链路。

这就是第一个问题。
不是 skill 没有，而是 skill 没被正确触发。

第二个层次，是依赖断裂。

哪怕 skill 都触发对了，顺序不对也没用。

很多工作流天然有依赖关系。
先看问题，再定范围。
先做 spec，再做计划。
先复现 bug，再写测试。
先通过验证，再允许发布。

这些不是写作风格问题，是工程约束问题。

如果没有依赖层，Agent 就会特别像一个很聪明但很急的人。
它会跳步骤。
它会把 review 当成修改。
它会把计划当成执行。
它会把结论当前提。

你会发现，很多人以为 AI agent 的问题是偷懒。
其实更准确一点，它的问题是没有流程秩序。

而 skill 编排，本质上就是在给这些局部能力加秩序。

第三个层次，是状态丢失。

很多 skill 单独看都没问题，真正一串起来就废了，原因通常不是某个 skill 写得差，而是前后状态没有被传好。

前一个 skill 产出的 spec，没有变成后一个 skill 的强约束。
测试 skill 跑出来的失败原因，没有进入修复 skill 的输入。
review skill 发现的风险，没有变成 ship skill 的阻断条件。

于是整个系统看起来很忙，实际上每一步都像重新开始。

这就是为什么 skill 数量越多，有时候反而越乱。

因为系统复杂度上升了，但协同机制没上升。

说白了，skill 数量解决的是覆盖面。
skill 编排解决的是完成率。

覆盖面高，只代表你理论上什么都能做一点。
完成率高，才代表你真的能把一个复杂任务做完。

## 怎么办

如果你真的想把 skill 编排做好，我觉得最少要补五层。

第一层，触发层。

不要再把 skill 触发理解成关键词匹配。
触发层应该判断的是任务阶段和任务意图。

用户现在是要理解问题，还是要解决问题。
是要生成方案，还是要执行方案。
是要探索，还是要收敛。
是要单点处理，还是要多步联动。

同样一句帮我优化这个页面，在不同阶段调用的 skill 可能完全不一样。
如果触发层做不好，后面全是偏的。

所以第一个实操建议就是，skill 不要只写 when to use，还要写 when not to use，以及它前后通常接什么 skill。

第二层，依赖层。

每个 skill 都应该有自己的前置条件和后置产物。

前置条件决定它什么时候能安全启动。
后置产物决定它给下游留下什么。

比如一个 spec skill 的后置产物，不应该只是输出一段文字，而应该是明确的问题定义、范围边界、成功标准。这样 plan skill 才能接。

再比如一个 test skill 的后置产物，不应该只是跑了测试，而应该是失败用例、复现路径、影响范围。这样 fix skill 才能接。

很多 skill 不能编排，不是因为不够聪明，而是因为输入输出没有被定义成可衔接的结构。

第三层，顺序层。

你得明确哪些 skill 可以并行，哪些必须串行。

有些东西天然能并行。
比如多个 reviewer 从不同维度看同一段代码。
比如 research 和素材收集一起做。
比如多个子任务拆给不同 agent。

但有些东西不能并行。
比如没做 spec 就直接 build。
比如没过 verify 就直接 ship。
比如 plan 和 execute 一边生成一边互相覆盖。

所以顺序层的关键，不是让系统更快，而是让它知道什么地方不能快。

复杂任务里，速度不是第一优先级。
顺序正确才是。

第四层，角色层。

一旦 skill 多起来，单 Agent 往往不够用。
这时候真正需要的不只是更多 skill，而是不同角色。

谁负责拆任务。
谁负责执行。
谁负责验证。
谁负责收口。
谁只做质检，不参与生产。

你会发现，一旦角色不分，Agent 特别容易自说自话。
它既提方案，又做执行，又给自己打分，最后很难客观。

所以很多编排最后都要走向角色分离。
不是为了炫 multi-agent，而是为了减少单个 Agent 又当运动员又当裁判的偏差。

第五层，校验层。

这是最容易被忽略的一层，但其实最关键。

skill 编排如果没有校验层，就只是把一堆动作串起来。
它不是真正的工作流。

真正的工作流一定要回答一个问题。
上一阶段的结果，凭什么被允许进入下一阶段。

比如 spec 有没有遗漏关键边界。
计划有没有过度拆分。
代码有没有通过测试。
修改有没有引入新风险。
发布条件有没有满足。

如果没有这些 gate，skill 编排最后只是更复杂的流水线错误传播器。

所以我现在越来越觉得，判断一个 skill 系统有没有成熟，不要看它有多少 skill。
看它有没有这些编排层。

能不能正确触发。
能不能表达依赖。
能不能控制顺序。
能不能分配角色。
能不能设置校验。

这五层里，少一层都能跑。
但跑不稳。

说到底，skill 这件事现在最容易被误解的地方，是大家都在把它当能力收藏。

今天装一个，明天再装一个。
像给 Agent 买装备。

但复杂任务不是装备战。
复杂任务是协同战。

真正拉开差距的，不是你仓库里有多少个 skill。
而是这些 skill 之间有没有被组织成一个有秩序的执行系统。

skill 的上限，是把一个点做对。
编排的上限，是把一条链路做通。

而后者，才决定 Agent 到底能不能从会一点，走到真的能交付。
